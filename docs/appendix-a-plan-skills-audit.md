> Appendix A of the [repository docs](../README.md#repository-docs).

> **Status: Historical — largely implemented.** This is a point-in-time audit; its file:line citations describe the skills as they were when it was written, and most of its recommendations have since shipped:
> finding 1.1 in [e4b9f48](https://github.com/awinogradov/code-assistants/commit/e4b9f48), 1.2 in [1835f85](https://github.com/awinogradov/code-assistants/commit/1835f85), 3.4 in [162791e](https://github.com/awinogradov/code-assistants/commit/162791e), 4.1 in [4825b20](https://github.com/awinogradov/code-assistants/commit/4825b20), 5.1 in [65fc291](https://github.com/awinogradov/code-assistants/commit/65fc291), and 7.1 in [8734148](https://github.com/awinogradov/code-assistants/commit/8734148).
> Read it for the audit method and the rationale behind the current skill structure — not as actionable guidance.

A dimension-by-dimension audit of the three planning skills, with each finding citing an exact file and line, rated by impact, and paired with a concrete recommendation. The deliverable is this findings document and the prioritized optimization plan at the end; no skill is rewritten here. The recommendations seed follow-up issues.

Tracking issue: [#171](https://github.com/awinogradov/code-assistants/issues/171).

## Methodology

Each skill and agent was read in full. Every claim below cites `file:line` against the files as they exist at audit time. Citations were then re-verified line-by-line by an independent pass; the corrections that pass produced are folded in here, and the ones worth recording are listed under [Citation verification notes](#citation-verification-notes). Impact is rated **High / Med / Low** by how much the issue inflates tokens, breaks control flow, or risks silent drift.

### 3. Context-gathering effectiveness

This is the dimension with the largest practical waste, and the framing matters — see the [Citation verification notes](#citation-verification-notes) for one claim that the verification pass corrected.

| #   | Impact | Finding                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           | Citation                                                                                                                                                    |
| --- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 3.1 | High   | **The same tree is traversed up to three ways in one run, with no stated division of labor.** (a) Phase 0 acquires a repomix snapshot and searches it via `grep_repomix_output`/`read_repomix_output`; (b) stack Phase 1 step 2 launches Explore agents **unconditionally**, which crawl the live tree via Grep/Glob/Read; (c) stack Phase 1 step 6 searches the snapshot again, and Phase 2 adds a "Fall back to Grep, Read, or Glob" live path. Nothing says which source answers which question, so broad discovery work is repeated.                                                                                                                                          | snapshot `plan/SKILL.md:104-141`; Explore agents `plan-bun/SKILL.md:43-48`; snapshot re-search `plan-bun/SKILL.md:56`; live fallback `plan-bun/SKILL.md:64` |
| 3.2 | Med    | **The snapshot-first contract is never propagated to the stack skills.** `docs/09-repomix-pack.md:53-61` defines a snapshot-first-with-fallback contract and the orchestrator's Phase 0 follows it, but the stack skills launch live Explore agents with no reference to it. The contract also states (`:61`) that on a feature branch the snapshot "lags by the in-flight changes — acceptable, since review-time skills obtain the actual diff separately." The plan stack skills _do_ obtain the branch diff separately (`plan-bun/SKILL.md:40-42`), so the unconditional Explore crawl is largely redundant with already-available context rather than a freshness necessity. | contract `docs/09-repomix-pack.md:53-61`; feature-branch lag `:61`; branch diff `plan-bun/SKILL.md:40-42`                                                   |
| 3.3 | Med    | **No snapshot freshness signal.** Phase 0 attaches `.repomix/pack.xml` whenever it exists, with no check of how stale it is (it is refreshed only on merge to `main` by `.github/workflows/repomix-pack.yml`). Combined with 3.1, the run can search a stale snapshot _and_ read fresh files with no rule for reconciling the two.                                                                                                                                                                                                                                                                                                                                                | `plan/SKILL.md:104-110`; `.github/workflows/repomix-pack.yml` (`pull_request: types:[closed]`)                                                              |
| 3.4 | Med    | **Doc-lookup / repo-docs / CLAUDE.md blocks are duplicated into the stack skills.** The orchestrator declares these under "Common Instructions … apply to ALL stacks," then each stack skill restates them in Phase 1 steps 3–5.                                                                                                                                                                                                                                                                                                                                                                                                                                                  | orchestrator `plan/SKILL.md:203-235` (lookup), `:237-244` (repo docs), `:271-273` (CLAUDE.md); duplicated `plan-bun/SKILL.md:49-55`                         |

Recommendation for the dimension: state one decision rule — snapshot serves broad/whole-repo reads; live tools (Explore/Grep) serve fresh-or-targeted reads the snapshot cannot — make the Explore step conditional on that rule, and drop the duplicated global blocks from the stack skills.

## Citation verification notes

Corrections produced by the independent re-verification pass, recorded so future readers trust the line numbers:

- **`docs/09-repomix-pack.md` contract location and meaning.** The snapshot-first-with-fallback contract is at `:53-61`, not earlier in the file. Importantly, the contract's fallback is to `pack_codebase` (still a pack) only when the pack is **absent or attach fails** — it is **not** a staleness-triggered fallback to live tools. Snapshot staleness on feature branches is explicitly called "acceptable" (`:61`). Finding 3.2 is framed accordingly; an earlier draft that described a "fall-back-when-stale" clause was incorrect.
- **`plan-bun` verify example line.** The `bun test` verify example is at `:156`. The corresponding `vitest run` example in `plan-nodejs-react` is at `:161` — the offset is real and caused by the longer (10-row) expert table shifting content down, which is itself evidence for finding 8.1.
- **`TaskCreate` grant line.** In `plan-bun`, `TaskCreate` is at `:8` (`TaskList` `:6`, `TaskUpdate` `:7`). This is the todo tool, distinct from the absent sub-agent launcher `Task` discussed in 4.1.
