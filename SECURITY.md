# Security Policy

We take the security of this project and the people who use it seriously. Thank you for helping keep it and its users safe.

## Supported Versions

Security fixes are applied to the latest released version and the default branch. Before reporting, please update to the latest version and confirm the issue still reproduces. Older releases are not back-patched — upgrading is the supported remediation.

## Reporting a Vulnerability

Please report security vulnerabilities privately. Do not open a public issue, pull request, or discussion for a suspected vulnerability — public disclosure before a fix is available puts every user at risk.

Use GitHub's Private Vulnerability Reporting:

1. Open the repository's **Security** tab.
2. Click **Report a vulnerability** to open a private draft advisory visible only to you and the repository's security maintainers.
3. Fill in the report (see _What to Include_ below) and submit.

If you do not see a **Report a vulnerability** button, Private Vulnerability Reporting is not enabled on this repository — a copied or forked repository does not inherit it from upstream. In that case, contact the repository owner or organization privately through a channel listed on their GitHub profile or repository homepage, rather than opening a public issue, pull request, or discussion.

## What to Include

A good report helps us triage quickly. Where possible, include:

- A description of the vulnerability and its impact.
- The affected component, file, or version.
- Step-by-step instructions to reproduce, including any proof-of-concept.
- Any known mitigations or workarounds.

## Response Expectations

This is a maintainer-run project, so triage is best-effort rather than bound by a formal SLA. You can expect:

- An acknowledgement of your report, typically within a few business days.
- An assessment of validity and severity, and follow-up questions if we need more detail.
- Progress updates as we work toward a fix, and credit for your report once a fix ships (unless you prefer to remain anonymous).

## Disclosure Policy & Scope

We follow coordinated disclosure: please give us a reasonable window to release a fix before any public disclosure, and we will work with you on timing.

In scope: the source code in this repository and the artifacts it publishes. Out of scope: vulnerabilities in third-party dependencies (report those upstream) and findings that require privileged access or social engineering. There is currently no paid bug-bounty program.
